# 第 1 步：开源项目调研与重构基线

## 目标

本阶段不继续修补旧代码，而是先研究成熟的开源单词学习/闪卡/SRS 项目，提炼出适合本项目的底层架构基线。

本次调研重点回答 5 个问题：

1. 成熟产品的数据模型怎么设计？
2. 学习调度和学习内容是否解耦？
3. 进度同步、学习日志、统计分析如何做？
4. 前后端如何拆分，才能便于后续扩展？
5. 哪些产品思路适合“单词学习网站”，哪些不适合？

## 调研对象

### 1. Anki

- 仓库：https://github.com/ankitects/anki
- 定位：成熟度最高的开源闪卡/SRS 生态
- 适合借鉴：
- `Deck / Note / Card / ReviewLog` 这种分层数据模型
- 学习调度引擎独立于 UI 和内容层
- 同一条词汇可以生成多张不同训练目标的卡片
- 不适合直接照搬：
- 桌面优先，系统偏重，不适合直接作为轻量网页产品模板

### 2. LibreLingo

- 仓库：https://github.com/LibreLingo/LibreLingo
- 定位：偏课程式语言学习平台
- 适合借鉴：
- 课程、单元、练习内容数据化配置
- 上层技能树/关卡结构清晰
- 适合把考试词库组织成可持续扩展的学习路径
- 不适合直接照搬：
- 如果只做背单词，课程系统可能太重

### 3. LinguaCafe

- 仓库：https://github.com/simjanos-dev/LinguaCafe
- 定位：阅读驱动的语言学习平台
- 适合借鉴：
- 阅读内容 -> 生词 -> 复习 的闭环
- 单词状态机设计，如未见、已见、学习中、掌握
- 内容驱动获词，不只是“刷卡”
- 不适合直接照搬：
- 需要较完整的内容系统，首版实现成本更高

### 4. Learning with Texts

- 仓库：https://github.com/HugoFara/lwt
- 定位：基于文本上下文的学词工具
- 适合借鉴：
- 在语境里学词，而不是孤立背词
- 词组、短语和上下文示例的重要性
- 数据模型相对简单，容易理解核心闭环
- 不适合直接照搬：
- 交互和界面比较老，不适合作为现代 UI 参考

### 5. Mnemosyne

- 仓库：https://github.com/mnemosyne-proj/mnemosyne
- 定位：经典开源 SRS 闪卡系统
- 适合借鉴：
- 最小可行复习系统如何收敛
- 答题结果、调度、统计之间的关系
- 不适合直接照搬：
- Web 化较弱，更适合作为调度内核参考

### 6. FSRS / Open Spaced Repetition

- 组织：https://github.com/open-spaced-repetition
- 资源集合：https://github.com/open-spaced-repetition/awesome-fsrs
- 定位：现代开源间隔重复调度算法与实现生态
- 适合借鉴：
- 把调度算法做成独立模块
- 后续可从简单规则升级到 FSRS
- 可优先考虑 TypeScript 实现，便于与 Web 项目集成
- 不适合直接照搬：
- 它解决的是调度问题，不解决完整产品形态

## 关键结论

### 结论 1：不能再把“单词数据、学习状态、页面状态”混在一起

旧项目最大问题不是页面丑，而是模型太浅。成熟项目都会把以下几层拆开：

- 内容层：词库、词书、例句、音频、标签、难度
- 学习对象层：卡片、题型、学习单元、复习队列
- 学习记录层：每次作答日志、耗时、正确率、复习间隔
- 用户进度层：某课程学到哪里、哪些已掌握、哪些待复习
- 页面会话层：当前停留在哪张卡、是否翻面、当前输入内容

本项目后续必须采用这种分层，不能再靠前端页面状态“顺便充当进度系统”。

### 结论 2：必须采用 `词条 -> 卡片 -> 复习记录` 的三层结构

建议核心模型如下：

- `VocabularyItem`
  - 单词、音标、释义、词根、例句、主题、等级、音频
- `CardTemplate`
  - 学习模式，如识义、拼写、听写、例句填空、反向回忆
- `LearningCard`
  - 某词条按某模板生成的一张卡
- `ReviewLog`
  - 每次作答的结果、时间、耗时、评分、下次复习时间
- `StudyDeck`
  - 一个词书、课程或考试范围的集合

这样做的好处是：

- 一个单词可以生成多种训练方式
- 后期加新题型不需要重建词库
- 统计和推荐有真实数据基础

### 结论 3：学习调度必须独立成服务或独立模块

不能把“解锁下一单元、是否该复习、下一张卡是什么”散在页面组件里。

后续要单独做：

- `scheduler`
  - 决定今日新词数、复习数、卡片顺序
- `review-engine`
  - 根据作答结果更新卡片状态
- `progress-service`
  - 负责课程解锁、进度同步、统计口径

首版可以先用简单规则：

- 新词卡与复习卡分队列
- 错题优先回放
- 单元完成后才解锁下一单元

二期再升级到 FSRS。

### 结论 4：前后端必须彻底分层，不能继续“前端决定真相”

建议改成下面的结构：

- `apps/web`
  - 只负责 UI、交互、播放音频、渲染学习流
- `apps/api`
  - 只负责鉴权、课程、卡片、进度、日志接口
- `packages/domain`
  - 领域模型与业务规则
- `packages/scheduler`
  - 复习调度算法
- `packages/content`
  - 词库、词书、课程内容解析

这比现在前端和后端各写一点业务逻辑的方式更稳。

### 结论 5：产品形态要从“翻卡 demo”升级为“学习闭环”

成熟产品都不是只有翻卡，它们至少有下面这个闭环：

1. 获取学习内容
2. 进入学习队列
3. 作答与反馈
4. 记录日志
5. 更新掌握度与下次复习时间
6. 在仪表盘展示真实统计

本项目后续至少要支持：

- 新词学习
- 复习模式
- 听写模式
- 例句理解
- 错词回炉
- 真实统计与待复习清单

## 对本项目的直接重构建议

### 必须保留的方向

- 还是做单词学习网站
- 保留考试词库路线：中考 / 高考 / IELTS / TOEFL
- 保留 Web 优先，兼容手机浏览器

### 必须放弃的旧设计

- 页面组件直接承担核心业务逻辑
- 用户进度只存一个“解锁到第几单元”
- 词库只是一坨前端静态数组
- 学习完成后才临时同步，缺少细粒度学习日志

### 新架构基线

- 内容中心：词条、例句、音频、主题、词书
- 学习中心：卡片生成、学习队列、题型模板
- 调度中心：新学/复习/错题回放
- 账号中心：用户、会话、权限、设备同步
- 统计中心：学习日志、掌握率、待复习、热词、错词

## 建议的 4 步重构路线

### 第 1 步：开源调研与重构基线

- 已完成
- 产出：本文件

### 第 2 步：重做领域模型和项目骨架

- 定义数据库模型
- 定义 API 边界
- 拆分 monorepo 目录
- 搭建新的前后端和共享包结构

### 第 3 步：重建核心学习闭环

- 用户系统
- 词书和课程系统
- 卡片生成与学习队列
- 复习调度与学习日志
- 真实进度同步

### 第 4 步：重建前端体验与内容系统

- 学习页 UI/动效
- 仪表盘、复习页、错词页
- 音频、听写、跟读
- 内容管理与词库扩展

## 本阶段最终结论

如果这次真要“从底层架构全部推倒重来”，那新的项目不应该再叫“一个页面上翻几张卡片的网站”，而应该是：

一个以 `词条内容 + 卡片系统 + 调度引擎 + 学习日志 + 多端同步` 为核心的单词学习平台。

下一步就该进入第 2 步：正式重做领域模型、数据库结构和项目骨架。
